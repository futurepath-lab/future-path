# 第一阶段：推荐逻辑可靠化

这个阶段的目标不是把网站做得更炫，而是让推荐结果更可信、更可解释、更容易持续改进。

## 目标

1. 推荐结果能区分相近职业，而不只是给出泛泛的大方向。
2. 每个职业、城市、专业推荐都有可解释依据。
3. 用户可以反馈“不准”的原因，系统后续能据此调整。
4. 内容数据可以持续扩展，不需要每次都重写页面代码。
5. 明确说明网站是探索工具，不是替用户做最终决定。

## 推荐系统应包含的维度

### 用户侧信号

- 兴趣主题：技术、商业、创意、公共服务、教育健康、城市与国际等。
- 能力倾向：研究分析、沟通协作、创意表达、技术工具、规划执行。
- 工作偏好：面对人、面对系统、面对规则、面对内容、面对现场。
- 现实约束：居住城市、迁移意愿、生活成本敏感度、稳定性需求。
- 成长方式：长期训练、项目实践、作品驱动、考试筛选、访谈验证。

### 职业侧数据

- 职业族群：技术产品、商业运营、创意内容、教育健康、公共政策、法律风控、城市产业等。
- 典型岗位：入门岗位、进阶岗位、相关岗位。
- 能力权重：该职业真正依赖哪些能力。
- 学习门槛：专业基础、工具基础、资格要求、长期训练成本。
- 劝退点：高压力、强竞争、收入波动、准入门槛、城市限制。
- 入门证据：适合高中生/大学生先做的小项目。

### 城市侧数据

- 产业结构：科技、金融、文旅、制造、教育、公共服务等。
- 岗位密度：该方向在该城市是否有足够机会。
- 生活约束：成本、节奏、语言、签证、通勤、离家距离。
- 适合人群：偏稳定、偏创新、偏国际化、偏生活质量等。

### 专业侧数据

- 主攻专业：最直接连接职业的专业。
- 辅修专业：能增强竞争力的组合。
- 补充了解：降低试错成本的跨学科知识。
- 大学准备路径：大一到大四可以如何积累课程、项目和实习。

## 第一阶段优先级

### P0：已经完成或正在完成

- 增加职业分化题，把测试从 24 题扩展到 34 题。
- 将结果拆成职业匹配度、地区适配度、学习可行度。
- 增加推荐解释、职业对比、7 天行动计划。
- 城市详情只开放已有真实图片的城市，避免图片和城市不匹配。

### P1：下一步应做

- 给每个职业增加结构化字段：职业族群、劝退点、适合证据、学习门槛。（已开始：当前由 `getStructuredCareerProfile` 统一生成）
- 给每个城市增加结构化字段：岗位密度、生活成本等级、产业强度。（已开始：当前由城市画像启发式生成，作为调研线索而非官方统计）
- 让“不感兴趣”“地点太远”“门槛太高”等反馈被记录成可分析数据。（已开始：当前写入本地 `futurePathFeedbackLog`，可导出 JSON）
- 增加“推荐方法说明”页面，让用户知道分数怎么来的。

### P2：正式上线前应做

- 将职业库、城市库、专业库从代码中拆出去，变成 JSON 或数据库。（已建立迁移方案和 schema 草案）
- 增加内容审核表，标记哪些信息来自真实数据、哪些是经验判断。
- 做 20-50 人小规模测试，记录推荐准确率和用户困惑点。（已建立测试模板和记录表）
- 根据反馈重调职业权重和题目措辞。

## 推荐结果的可信边界

网站应该明确告诉用户：

- 测试结果是探索建议，不是性格定型。
- 职业推荐不能替代真实岗位调研、实习体验和专业咨询。
- 城市推荐需要结合最新产业、政策、签证、成本和家庭情况。
- 大学专业建议是路径参考，不是唯一入口。

## 衡量第一阶段是否完成

1. 用户能看懂为什么推荐某职业。
2. 用户能比较前 3 个职业的差异。
3. 用户能拿到一个可执行的 7 天验证计划。
4. 每个推荐至少有职业、城市、学习三类解释。
5. 网站有清晰的推荐方法说明。

## 第一阶段当前状态

第一阶段已形成可继续扩展的基础：

- 网站中已有推荐解释、职业对比、城市详情、行动计划和反馈日志。
- 数据层已有 schema、sample JSON、校验脚本和静态预览页。
- 测试层已有 20-50 人小规模测试模板。

下一阶段重点应转向：让页面逐步读取 JSON 数据、引入真实岗位/城市数据、准备上线架构。
